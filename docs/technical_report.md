# 1. Introduction

This report presents a multi-faceted analytics system for an e-commerce platform built on three complementary database technologies: **MongoDB** (document store), **Apache HBase** (wide-column store), and **Apache Spark** (distributed processing engine).

The system processes a synthetic dataset spanning **90 days** of e-commerce activity:

| Metric | Value |
|--------|-------|
| Users | 2,000 |
| Products | 800 |
| Categories | 15 |
| Browsing Sessions | 20,000 |
| Transactions | 4,390 |
| Total Revenue | $1,925,085,008 |

Each technology was chosen for its strengths: MongoDB for document-oriented queries on structured business entities, HBase for time-series and sparse columnar data, and Spark for cross-system batch processing and SQL analytics. The project covers four parts: (1) Data Modeling & Storage, (2) Spark Processing, (3) Analytics Integration, and (4) Visualization & Insights.

# 2. System Architecture

## 2.1 Data Flow

![Data Flow Architecture](data_flow.svg){ width=55% }

Raw data is generated by `dataset_generator.py` using the Faker library. The seeding script loads, denormalizes, and inserts data into both databases with appropriate indexing.

## 2.2 Service Topology

All infrastructure runs via Docker Compose with six services:

| Service | Image | Port | Role |
|---------|-------|------|------|
| MongoDB | `mongo:7.0` | 27017 | Document store |
| ZooKeeper | `zookeeper:3.9` | 2181 | HBase coordination |
| HBase | `dajobe/hbase` | 9090, 16010 | Wide-column store |
| Spark Master | `apache/spark:4.0.2-python3` | 7077, 8080 | Cluster manager |
| Spark Worker | `apache/spark:4.0.2-python3` | 8081 | 2GB RAM, 2 cores |
| Dashboard | `Dockerfile.dashboard` (Streamlit) | 8501 | Interactive analytics UI |

HBase runs natively on x86_64 (WSL/AMD). On Apple Silicon, `platform: linux/amd64` emulation is used as a fallback. Spark processing uses local mode (`local[*]`) with Python drivers (`pymongo`, `happybase`). A WSL-optimized compose override (`docker-compose.wsl.yml`) provides an `init` service that auto-generates data and seeds both databases on startup.

## 2.3 Technology Justification

- **MongoDB**: Natural fit for hierarchical product catalogs (embedded categories), user profiles (embedded purchase summaries), and transactions (embedded line items). The aggregation framework enables complex analytical queries without ETL.
- **HBase**: Ideal for time-series session data with sparse columns (varying page views, cart contents) and product metrics requiring efficient range scans by composite row keys.
- **Spark**: Compute layer for cross-system joins (MongoDB + HBase data), batch analytics (co-purchase recommendations, cohort analysis), and complex SQL queries spanning multiple sources.

# 3. Part 1: Data Modeling & Storage

## 3.1 MongoDB Schema Design

Three collections with **7 indexes** total:

**Products** (800 docs) --- Each product embeds full category/subcategory info, denormalized from the 15-record categories dataset to eliminate `$lookup` joins:

```json
{
  "_id": "prod_00001",
  "name": "Quality-focused web-enabled superstructure",
  "category": {
    "category_id": "cat_01", "name": "Electronics",
    "subcategory_id": "sub_001",
    "subcategory_name": "Smartphones",
    "profit_margin": 0.25
  },
  "base_price": 130.84,
  "current_stock": 143, "is_active": true,
  "price_history": [{"date": "...", "price": 130.84}]
}
```

Indexes: `category.category_id`, `base_price`, `is_active`.

**Users** (2,000 docs) --- Embeds pre-computed `purchase_summary` (total orders, total spent, avg order value, first/last purchase, favorite payment method). Indexes: `geo_data.province`, `purchase_summary.total_spent`, `registration_date`.

**Transactions** (4,390 docs) --- Embeds `items` array with line items. Indexes: `user_id`, `timestamp`, `status`, `items.product_id`.

**Design decisions**: Categories are *not* a separate collection (only 15 records; denormalized into products). Sessions are *not* in MongoDB (sparse columns and time-series access patterns suit HBase better).

## 3.2 HBase Schema Design

Two tables with **6 column families**:

**Sessions** (20,000 rows) --- Row key: `{user_id}_{reverse_timestamp}` where reverse timestamp = `9999999999999 - epoch_ms`, ensuring newest-first ordering on prefix scans. Column families: `info` (session metadata), `device` (type/os/browser), `geo` (city/state/country), `activity` (JSON-serialized page views, cart, viewed products).

**Product Metrics** (2,274 rows) --- Row key: `{product_id}_{date}`. Column families: `sales` (quantity, revenue, transactions, avg price), `inventory` (stock, base price, active status).

# 4. Part 2: Spark Processing Pipelines

## 4.1 Data Cleaning & Normalization

All JSON datasets loaded with `multiLine=True` and Spark schema inference. Cleaning steps:

- Null `current_stock` filled with 0; null `is_active` filled with `True`
- String timestamps cast to `TimestampType`
- Primary key uniqueness verified (0 duplicates across all entities)
- Subcategories exploded into flat lookup table via `F.explode()`

## 4.2 Product Recommendations

Co-purchase analysis via self-join on transaction items:

1. Explode items to (transaction\_id, product\_id) pairs
2. Self-join on transaction\_id with filter `product_a < product_b`
3. Count co-occurrences; top pair: prod\_00085 + prod\_00213 (3 co-purchases)
4. Window function (`ROW_NUMBER`) selects top-3 recommendations per product

## 4.3 Cohort Analysis

Users grouped by registration month (24 cohorts from Dec 2023 to Nov 2025). Key findings:

- Largest cohort: ~80 users per month with proportional revenue
- Retention rates: ~40--60% of users transact in their first month post-registration
- Spending concentration: most revenue occurs 1--3 months after registration

## 4.4 Spark SQL Queries

Five analytical SQL queries were executed:

**Query 1 --- Top Revenue Products**: Highest-grossing product is "Monitored interactive synergy" ($16,967 revenue, $2,715 estimated profit at 16% margin). Top 15 products span Jewelry, Health, Home & Garden, and Sports categories.

**Query 2 --- Session Behavior by Device**: Desktop/Windows has highest session count (993) with 23.2% conversion. Tablet/iOS converts best at 24.7%. Average session duration ranges 1,738--1,888 seconds across device/OS combinations.

**Query 3 --- Conversion Funnel**: 20,000 total sessions, 14,734 added to cart (73.7%), 4,390 converted (21.9%). Cart-to-purchase rate: 29.8%.

**Query 4 --- Daily Revenue with 7-Day Moving Average**: The 7-day moving average smooths daily fluctuations. Cumulative revenue reaches $1,925,085,008.

**Query 5 --- Price Elasticity**: All products show zero price standard deviation (synthetic data uses fixed pricing), so elasticity ratios are NULL. The query structure (3 CTEs with STDDEV) demonstrates the analytical approach for real-world variable pricing.

# 5. Part 3: Analytics Integration

## 5.1 Customer Lifetime Value (CLV)

**Business Question**: *"What is the estimated lifetime value of each customer segment, and how does engagement correlate with spending?"*

**Data Sources**: MongoDB `users` (profiles), MongoDB `transactions` (purchase history), HBase `sessions` (browsing engagement).

**Method**: Cross-system join in Spark on `user_id`. CLV = `avg_order_value * annualized_purchase_frequency * 2.0` (estimated lifespan). Segmented into four tiers.

**Results**:

| CLV Tier | Customers | Avg CLV | Avg Sessions | Avg Conv Rate |
|----------|-----------|---------|--------------|---------------|
| Platinum | 329 | $104,895 | 10.9 | 29.1% |
| Gold | 42 | $3,140 | 8.8 | 12.6% |
| Silver | 55 | $1,308 | 8.5 | 13.6% |
| Bronze | 74 | $47 | 8.0 | 1.5% |

**Key finding**: Platinum users average **10.9 sessions** vs 8.0 for Bronze, and convert at **29.1%** vs 1.5%. Higher engagement directly correlates with lifetime value. Kigali City has the highest average CLV ($134,604), followed by Northern Province ($105,289).

## 5.2 Funnel Conversion Analysis

**Business Question**: *"What is the conversion rate at each funnel stage, and which factors influence conversion?"*

**Data Sources**: HBase `sessions` (browsing, cart, conversion, device), MongoDB `transactions` (purchase amounts).

**Overall Funnel**:

| Stage | Count | % of Total |
|-------|-------|------------|
| Browse | 20,000 | 100.0% |
| View Product | 20,000 | 100.0% |
| Add to Cart | 14,734 | 73.7% |
| Purchase | 4,390 | 21.9% |

**Cart abandonment**: 10,344 sessions (70.2%) added items to cart but did not purchase.

**By device**: Mobile converts best at 22.5%, desktop at 22.1%, tablet at 22.1%. Abandonment rates are uniform across devices (~69.5%).

**By referrer**: Direct and search engine traffic convert at 22.4%, email at 22.4%, social media at 22.1%, affiliate at 21.7%.

**By time of day**: Afternoon sessions convert best (22.9%), followed by night (22.3%), evening (21.9%), and morning (21.7%).

# 7. Scalability Considerations

**MongoDB**: Horizontal scaling via sharding (shard key on `user_id` or `category_id`). Read scaling through replica sets. The embedded document model minimizes cross-shard joins.

**HBase**: Automatic region splitting distributes data as tables grow. The `user_id_reverse_timestamp` row key distributes evenly across regions while preserving per-user locality. Column family separation enables selective reads.

**Spark**: Horizontal compute scaling by adding workers. Partition tuning via `spark.sql.shuffle.partitions`. Can co-locate with HBase RegionServers for data locality.

**Current limitations**: Single Docker host (no distributed deployment), Spark in local mode. HBase runs natively on x86_64/WSL; Apple Silicon requires x86 emulation.

# 8. Limitations & Future Work

**Limitations**:

- *Synthetic data*: Generated with Faker, producing uniform distributions without real-world seasonality or anomalies. The price elasticity analysis (Query 5) yields NULL ratios because synthetic pricing has zero variance.
- *HBase emulation*: On Apple Silicon, running `dajobe/hbase` (linux/amd64) via QEMU adds significant startup latency (5--10 minutes). Deploying on native x86_64 (WSL) eliminates this overhead. The analytics integration module includes a JSON fallback for when HBase is unavailable.
- *Single-node deployment*: Does not demonstrate true distributed processing.

**Future enhancements**:

- *Real-time streaming*: Spark Structured Streaming for live session event processing
- *Machine learning*: ALS collaborative filtering (Spark MLlib) for production recommendations
- *~~Interactive dashboard~~*: Implemented as a Streamlit app (`dashboard.py`, served on port 8501 via Docker). Live at https://dbms-fat.bahatijustin.dev
- *Change Data Capture*: MongoDB Change Streams to trigger incremental HBase updates
